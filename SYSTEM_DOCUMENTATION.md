# Complete System Documentation - Multi-Tenant Task Management System

## 🎯 System Overview

A comprehensive multi-tenant task management system with role-based access control, admin approval workflow, and company-based user organization.

### Key Features
- ✅ Multi-tenant architecture with company isolation
- ✅ Three-tier user hierarchy (Super Admin → Company Admin → Users)
- ✅ Admin approval workflow with email notifications
- ✅ Company code-based user registration
- ✅ Google OAuth + Email/Password authentication
- ✅ Task management with Kanban board
- ✅ Real-time notifications
- ✅ Hidden super admin access URL

---

## 👥 User Roles & Permissions

### 1. Super Admin
**Access**: `/system-admin-access` (hidden URL)

**Capabilities**:
- View all companies, admins, and users
- Approve/reject admin registration requests
- Create company admins directly
- Generate company codes
- View detailed company information
- Receive email notifications for new admin requests

**Dashboard Features**:
- Pending admin requests panel with approve/reject buttons
- Company overview table (clickable rows for details)
- Stats: Total companies, admins, pending requests
- Company details modal showing all admins and users

### 2. Company Admin
**Access**: `/admin/login`

**Capabilities**:
- Register with company name (requires super admin approval)
- View and share company code with employees
- Manage users within their company
- Create and assign tasks to company users
- View all company tasks and user activities

**Dashboard Features**:
- Company code display with copy button
- User management panel
- Task creation and assignment
- Company-wide task overview

### 3. Regular User
**Access**: `/login`

**Capabilities**:
- Register using company code (provided by admin)
- View assigned tasks
- Update task status (Not Started → Working → Stuck → Done)
- Add completion remarks
- Receive task notifications

**Dashboard Features**:
- Kanban board view
- Task list view
- Calendar view
- Analytics dashboard

## 🔐 Authentication & Authorization

### Registration Flows

#### Super Admin Setup
```bash
# One-time setup via script
node todo-multiuser-backend/scripts/setup-super-admin.js
```

#### Admin Registration
1. Navigate to `/admin/register`
2. Fill form: Name, User ID, Email, Password, Company Name
3. Status set to `pending`
4. Super admin receives email notification
5. Super admin approves/rejects from dashboard
6. Admin can login after approval

#### User Registration
1. Navigate to `/register`
2. Fill form: Name, User ID, Email, Password, **Company Code**
3. System validates company code exists
4. User assigned to company automatically
5. Immediate access after registration

### Login Flows

#### Super Admin Login
- URL: `/system-admin-access` (hidden)
- Credentials: Email + Password
- Redirects to super admin dashboard

#### Admin Login
- URL: `/admin/login`
- Credentials: User ID/Email + Password
- Blocks if status is `pending` or `rejected`
- Redirects to admin dashboard

#### User Login
- URL: `/login`
- Options: Email/Password OR Google OAuth
- Redirects to user dashboard

## 🏢 Company Management

### Company Code System

**Purpose**: Unique identifier for each company to organize users

**Generation**: 
- Auto-generated by super admin (8-character alphanumeric)
- Or manually specified during admin creation

**Usage**:
- Admins share code with employees
- Users enter code during registration
- System validates and assigns user to company

**Display**:
- Super admin: Sees all company codes in overview table
- Company admin: Sees their code in profile with copy button
- Users: Enter code during registration only

### Company Details View (Super Admin)

Click any company row to see:
- Company code
- List of all admins with status badges
- List of all users
- Email and user ID for each person

## 📧 Email Notification System

### Configuration
```env
MAIL_USER=your-gmail@gmail.com
MAIL_PASS=your-gmail-app-password
SEND_LOGIN_EMAILS=true
SUPER_ADMIN_EMAIL=super-admin@company.com
```

### Notification Types

1. **Admin Registration Request**
   - Sent to: Super admin
   - Trigger: New admin registers
   - Content: Admin details, company name, approval link

2. **Login Notifications** (Optional)
   - Sent to: User
   - Trigger: Successful login
   - Content: Login time, device, location

3. **Password Reset**
   - Sent to: User
   - Trigger: Forgot password request
   - Content: 6-digit OTP code (10-minute expiry)

## 🔒 Security Features

### Account Status System
- `active`: Full access granted
- `pending`: Awaiting approval (admins only)
- `rejected`: Access denied
- `incomplete`: Needs account completion (Google OAuth users)

### Login Blocks
```javascript
// Admin login checks
if (account_status === 'pending') {
  return "Account pending approval. Please wait for super admin approval."
}
if (account_status === 'rejected') {
  return "Account has been rejected. Contact support."
}
```

### Hidden URLs
- Super admin access: `/system-admin-access` (not `/superadmin/login`)
- Generic link on login page: "System Access"
- Prevents unauthorized discovery

### Password Requirements
- Minimum 6 characters
- Bcrypt hashing with salt rounds: 12
- Password reset with OTP verification

## 🗄️ Database Schema

### Users Table
```sql
CREATE TABLE users (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT UNIQUE NOT NULL,
  user_id TEXT UNIQUE,
  password TEXT,
  auth_provider TEXT DEFAULT 'local',
  account_status TEXT DEFAULT 'active',
  role TEXT DEFAULT 'user',
  company TEXT,
  is_super_admin BOOLEAN DEFAULT false,
  google_id TEXT,
  email_verified BOOLEAN DEFAULT false,
  reset_token TEXT,
  reset_token_expiry TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW()
);
```

### Key Fields
- `account_status`: Controls access (active/pending/rejected/incomplete)
- `role`: Defines permissions (user/admin)
- `company`: Company code for multi-tenancy
- `is_super_admin`: Super admin flag
- `auth_provider`: local/google/hybrid

## 🛣️ API Routes

### Authentication Routes

#### Public Routes
```
POST /api/auth/register              # User registration (requires company code)
POST /api/auth/login                 # User login
POST /api/auth/admin/register        # Admin registration (pending status)
POST /api/auth/admin/login           # Admin login (checks status)
GET  /api/auth/google                # Google OAuth initiate
GET  /api/auth/google/callback       # Google OAuth callback
POST /api/auth/forgot-password       # Request password reset
POST /api/auth/reset-password        # Reset password with OTP
POST /api/auth/complete-account      # Complete Google OAuth account
```

#### Protected Routes
```
GET  /api/auth/profile               # Get current user profile
GET  /api/auth/users                 # Get users (filtered by company)
POST /api/auth/change-password       # Change password
```

### Super Admin Routes
```
GET  /api/auth/superadmin/companies          # Get all companies with stats
GET  /api/auth/superadmin/company/:code      # Get company details
GET  /api/auth/superadmin/pending-admins     # Get pending admin requests
POST /api/auth/superadmin/admin-action       # Approve/reject admin
POST /api/auth/superadmin/create-company-admin # Create admin directly
```

### Task Routes
```
GET    /api/tasks/assignedTo/:userId    # Get tasks assigned to user
GET    /api/tasks/assignedBy/:userId    # Get tasks created by user
GET    /api/tasks/all                   # Get all tasks (admin only)
POST   /api/tasks                       # Create new task
PUT    /api/tasks/:id                   # Update task
PATCH  /api/tasks/:id                   # Update task status
DELETE /api/tasks/:id                   # Delete task
```

## 📊 Complete User Flows

### Flow 1: New Company Setup
1. Super admin creates company admin with company code
2. Admin gets approved by super admin
3. Admin logs in and sees company code in profile
4. Admin shares code with employees
5. Employees register using the code
6. Employees can immediately start using the system

### Flow 2: Admin Self-Registration
1. Admin navigates to `/admin/register`
2. Fills form with company name
3. System sets status to `pending`
4. Super admin receives email notification
5. Super admin reviews in dashboard
6. Super admin approves/rejects
7. Admin can login if approved

### Flow 3: User Task Management
1. User logs in to dashboard
2. Views assigned tasks in Kanban board
3. Drags task to "Working on it"
4. Completes task, moves to "Done"
5. Admin sees updated status
6. User receives completion notification
